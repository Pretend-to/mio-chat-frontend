# 项目技术重难点 FAQ (面试专用)

本文档总结了本项目中的核心技术挑战、解决方案及其背后的思考，旨在为面试提供参考。

---

## 一、 即时通讯与网络层 (Real-time & Networking)

### Q1: 在 WebSocket 这种长连接中，如何保证连接的稳定性？

**A:**

1. **多协议回退 (Transport Fallback)**: 项目采用了 Socket.IO，并手动实现了更细粒度的控制。初始化时强制先尝试 `websocket` 传输。如果连接失败或被拦截，会自动回退到 `polling` 轮询模式，确保在防火墙或复杂网络环境下依然可用。
2. **连接生命周期管理**: 监听 `connect_error` 和 `disconnect` 事件。对于非主动断开的情况，设置了合理的重连间隔（`reconnectionDelay`）和最大延迟，防止瞬时网络抖动导致连接永久丢失。
3. **心跳机制**: 利用 Socket.IO 内置的心跳维持长连接，并在前端通过 UI 指示器实时展示当前延迟（ms），增强用户感知。

### Q2: 既然使用了 WebSocket，为什么还要在上面封装一套 fetch (Request-Response) 机制？

**A:**
标准的 WebSocket 是基于事件推送的，是异步的双工通信。但在应用层，我们经常需要像 HTTP 一样发送请求并等待结果。
**解决方案**:

- 我们在 `Socket` 类中实现了一个 `fetch` 方法。
- 发送请求时生成唯一的 `request_id`，并将该 ID 存入 `pendingRequests` 集合。
- 利用 `EventEmitter` 监听同名的 `request_id` 事件。
- 后端在响应消息中携带相同的 `request_id`，前端收到后触发事件，从而使 `fetch` 方法的 Promise 得以 resolve。
- 同时设置了 60s 的超时机制，防止请求挂死。

## 二、 大文件上传与性能优化 (File Upload & Performance)

### Q3: 处理大文件上传时，如果网络不稳定或者文件极大，你是如何优化的？

**A:**
核心策略是 **“分片上传 + Web Worker 预处理”**:

1. **分片上传 (Chunking)**: 将大文件切分为 1MB 的小块并发/顺序上传。如果某一分片失败，只需重传该分片，极大提高了断点续传的潜力和成功率。
2. **非阻塞 MD5 计算**: 在正式上传前，需要计算文件的 MD5 值用于后端校验。为了避免在主线程计算大文件哈希导致页面卡顿（FCP/Long Tasks），我们将哈希计算逻辑放在了 **Web Worker** 中运行。
3. **进度精确控制**: 通过 `XHR.upload.onprogress` 监控每个分片的进度，并根据“当前分片索引/总分片数”计算出全局进度返还给 UI。

## 三、 数据持久化与状态管理 (Persistence & State Management)

### Q4: 聊天应用产生的数据量很大，你是如何处理前端存储性能问题的？

**A:**

1. **持久化引擎**: 弃用了容量受限（5MB）且同步阻塞的 `localStorage`，改用 **IndexedDB**（通过 `localforage` 库封装）。IndexedDB 是异步的，支持存储大数据量，不会阻塞主线程。
2. **防抖写入 (Debounced Storage)**: 聊天频率高时，频繁操作 IndexedDB 会有 I/O 开销。项目对持久化操作进行了 **Debounce（防抖）处理**，将短时间内的多次数据变更合并为一次磁盘写入，平衡了数据安全与运行性能。
3. **响应式与解耦**: 核心业务逻辑在 `Client` 类中维护，通过 Vue 的 `reactive` 包装需要展示的数据（如 `contactList`），实现逻辑与 UI 的高效联动。

## 四、 复杂 UI 与 交互优化 (Complexity & UX)

### Q5: 针对 LLM 的打字机输出效果，前端是如何高效渲染并保持滚动体验的？

**A:**

1. **流式渲染**: 监听 `llm_message` 事件，实时更新消息对象中的文本。配合专门开发的 `MdRenderer` 组件（支持 Markdown、Katex、Mermaid），在内容不断增长时进行局部重绘。
2. **智能滚动策略**:
   - **自动触底**: 当用户处于页面底部时，新消息到达会自动触发 `scrollToBottom`。
   - **用户锁定**: 如果用户手动向上滚动查看历史记录，系统会停止自动触底，并展示一个“返回底部”的浮标，避免干扰用户阅读。
   - **图片加载处理**: 监听图片的 `load` 事件，防止图片加载撑开高度后导致滚动位置错乱。

### Q6: 为什么要在项目中手动管理 Service Worker 和缓存清理？

**A:**
在频繁迭代的 Web App 中，Service Worker 的强缓存机制可能导致用户无法及时获得代码更新，或者 IndexedDB 中的老数据与新逻辑冲突。

- 我们在 `Client.resetCache` 中实现了显式的 SW 注销逻辑。
- 通过 `MessageChannel` 与 Service Worker 通信，实现指令式的缓存清理，确保在用户遇到异常或手动重置时，能够彻底还原应用状态。

## 五、 架构设计 (Architecture)

### Q7: 项目是如何支持多协议（如 OneBot 和直接调用 LLM）的？

**A:**
项目采用了 **适配器/抽象类设计思想**:

- `Client` 类作为总控，不直接关心具体协议细节。
- `Contactor` 类作为各平台的抽象，不同平台的机器人（OneBot、OpenAI 等）通过配置项（options）区分行为。
- 在消息处理链路中，根据消息头中的 `protocol` 字段分发给不同的处理器，实现了业务逻辑的解耦，方便未来扩展如飞书、钉钉等新平台。

## 六、 复杂编辑器与多模态输入 (Editor & Multimodal)

### Q8: 为什么聊天输入框不使用简单的 `<textarea>`，而是使用 `contenteditable`？

**A:**

1. **图文混排 (Mixed Content)**: 为了支持像微信/QQ 那样的体验，让用户可以在输入框内直接看到粘贴的图片、表情，甚至是文件卡片。`<textarea>` 只支持纯文本，无法满足多模态交互的需求。
2. **高度自适应**: `contenteditable` 的元素天然支持根据内容自动撑开高度。

### Q9: 使用 `contenteditable` 带来了哪些挑战，你是如何解决的？

**A:**

1. **光标丢失与定位**: 在插入表情或上传图片后，系统往往会失去焦点。我们通过手动管理 `Selection` 和 `Range` 对象，记录并还原光标位置，确保用户在插入多媒体内容后能继续无缝输入。
2. **粘贴内容清理 (Paste Sanitization)**: 默认粘贴会带入大量冗余的 HTML 样式。我们重写了 `onPaste` 事件，拦截剪贴板内容，区分文本和图片。文本部分进行纯净化处理再转为 HTML 插入，图片部分则触发异步上传。
3. **性能与兼容性**: `contenteditable` 在不同浏览器下行为不一。项目通过统一的 `insertHtmlAtCursor` 方法封装了底层的 DOM 操作，确保跨端行为一致。

### Q10: 既然支持图片粘贴，如何防止大图直接上传导致压力过大？

**A:**
项目实现了 **“客户端预压缩”** 策略：

- 当用户粘贴或拖入图片时，不直接发送二进制流。
- 而是利用 `HTML5 Canvas API` 在内存中重新绘制图片，根据配置进行质量压缩（如 JPEG 0.7）。
- 压缩后再转为 `Blob` 进行上传，由于大多数图片在客户端就被缩小了体积，极大地提升了上传速度和用户体验。

## 七、 离线化与进阶 PWA (Advanced PWA)

### Q11: 你们的 Service Worker 缓存方案有什么特别之处？

**A:**
项目没有使用常规的 `Cache API`（如 Workbox 默认方案），而是自研了一套 **基于 IndexedDB 的 Service Worker 缓存系统**:

1. **带 TTL 的缓存控制**: 标准的 `Cache API` 不支持过期时间。我们通过在 IndexedDB 中存储 `Response + Timestamp` 的结构，手动实现了 7 天有效期的 TTL 机制。
2. **请求归一化 (Normalization)**: 在生成缓存 Key 时，会自动排序 URL 的查询参数。这解决了同一个资源因为参数顺序不同（如 `?a=1&b=2` 和 `?b=2&a=1`）导致缓存失效的问题。
3. **分流策略**: 严格区分静态资源和 API 请求。API 请求采取“网络优先”，而静态资源则采取“缓存优先 + 定期异步清理”，极大提升了弱网环境下的首屏加载速度。
